Score Model Execution

In a score model, the values you enter in the bin columns are used to generate underlying rule conditions. When the score model executes, those rules resolve to a ruleset representing the score model. Incoming data is tested against the bins (rule conditions), and when a value matches a bin, a partial score is assigned to the characteristic. The value used for the partial score is specified in the Score cell of each bin. The score model returns an initial raw score that represents the sum of the partial scores for all of the model’s characteristics.

The following figure is an example of a score model with a characteristic called FICO® Score that contains several bins.



Bins showing Range and Score values

When the score model in this example is run, and the value for the applicant’s FICO® score is 720, the fourth bin is used and a partial score of 300 is assigned. The partial score of 300 is then added to any partial scores that may be assigned to other characteristics in the score model to create a raw score. The raw score is further adjusted and aligned to produce a final score. Optionally, you can return a reason code and reason message for each rule that fired. Reason codes provide meaningful feedback about each partial score and its contribution to the score result.

Note While partial scores are generally positive values, there are situations where negative values are assigned. Partial scores are typically assigned within each characteristic, but also relative to the impact the bin values have on the overall calculated score. The characteristic baseline is one way of attempting to normalize the impact of score weights across all characteristics, but negative partial scores can also be used to balance the effects of bin values within a set of characteristic values. For example, you could have a scenario in which some values of a particular characteristic (such as a person’s income when applying for a loan) might result in positive scores in most cases (meaning that the bin values produce a positive impact on the score); however, the modelers might decide that certain values should produce a negative impact instead and use a negative partial score to do so.
See Also The Decision Module comes with preconfigured reason codes (RC00-RC99) and messages (MESSAGE 00-MESSAGE 99). For instructions on how to customize reason codes and messages, see the FICO Origination Manager Decision Module Administrator's Guide.

In decision metaphors such as decision tables or decision trees, you set values for both rule conditions and actions. In a score model, however, you set the properties and values for the rule conditions; the rule action is then generated for you using the value you entered in the Score column.

Score models are decision flow step implementation types, and can be invoked directly by a decision flow. They can also be invoked by a scoring scenario. At the end of score model processing, the final score that is generated can be used by other components, such as rulesets, decision tables, decision trees, data methods, and scenarios. For more information on using score models in scenarios, see Scoring Scenarios.

Score models can replace many of the rules that you would normally create manually in one or more rulesets. Creating a score model allows the generation of an arbitrary number of rules while returning a score based on all of the characteristics.
