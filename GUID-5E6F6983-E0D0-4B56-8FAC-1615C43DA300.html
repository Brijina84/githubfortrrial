Data Object Reference Definition

Use the Data Object Reference Properties page to define data object reference context, input arguments, and conditions.

When you create a data object reference, the Decision Module assigns a default context in the object model for the data object reference. You can edit the data object reference properties to modify the properties.

Object Context

Data object references return an object or a list of objects in the object context. You can also add conditions to evaluate the fields in the object context or in the hierarchical context of the parent object. For example, if the object context is Application.creditApplication.personalApplicant, the data object reference returns a personalApplicant object.

Input Arguments

Data object references can accept input arguments. Use input arguments to create generic data object references that can be reused with different pararameter. Input arguments can be used to evaluate conditions. Data object references with inputs can be used only by data methods.

For example, you can create a data object reference called dor_personalApplicantType that accepts an input argument xInput1 with data type string. Configure the following condition in the data object reference: personalApplicant.type is equal to xInput1. A data method can call the data object reference personalApplicant.type("Primary"). The data object reference dor_personalApplicantType extracts personalApplicant objects where personalApplicant.type is "Primary".

Alternatively, a data method can call personalApplicant.type("Co-Applicant") to perform calculations on a different data set.

Conditions

Data object references can evaluate conditions to extract objects that match specific criteria. You can configure the data object reference to return objects when all the conditions are true or when any one is true.

The left side of the condition expression can accept fields in the object context or the hierarchical context of the parent object. The object context path is the sequence of object names, separated by periods, with the first object always being the Applicationobject. For example, if the object context is Application.creditApplication.personalApplicant, you can select fields in the objects Application, creditApplication, and personalApplicant from the object model browser.

Select a valid operator from the drop-down list based on the data type of the condition field on the left side of the condition expression.

The right side of the condition expression can accept the following user-input:
  • Input arguments. Parameters sent from a data method.
  • User input.
  • Enumerations. Select a predefined value available for certain fields in the object model.
Note DORs can fail to operate as expected due to incorrect XPath syntax when condition fields use schema elements rather than attributes. For example consider a DOR, dor_MyObjects that uses condition fields, field1, field2, and field3: 
<Object field1="hello" field2="there">
       <field3>world</field3>
     </Object>
Where "field1" is an attribute and "field3" is a schema element. In DOR conditions all fields are assumed attributes, so an incorrect XPath is generated for field3 such as /Object/@field3.

Change the schema to define the fields as attributes (xs:attribute) rather than elements (xs:element) or use the DOR in a Data Method that evaluates the condition for field3.

Instead of using the following Data Method: 
for every Application.Object in dor_MyObjects do
//do something
end for
Use the following Data Method: 
for every Application.Object in dor_MyObjects do
if (field3="world") then
//do something
end if
end for
