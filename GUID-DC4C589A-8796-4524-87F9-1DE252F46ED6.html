Testing Data Methods: Overview

By changing a data method expression's conditional logic and editing the XML input against which the data method is run, you can test various scenarios for multiple data sets directly from the Strategy Management Workstation.

With the Test feature, data method development time is significantly reduced. It is particularly efficient when testing data methods with dependencies (a data method that depends on a data method that depends on another) because all dependent data methods are compiled and run at the same time.

When you select a data method and click the Test icon on its properties page, the Test Data Method window appears. The Create Test Data area is populated with an XML template that is generated depending upon the context of the data method. For example, if the context of the data method is Application.creditApplication.personalApplicant, you will see the template shown in the following figure. This data is a full XML payload that is valid for the XML schema that depicts your object model, starting at the root element (Application). You can edit the data to suit your test. Alternatively, you can use a base application (XML input application) to test the data method. When you select a base application, the corresponding XML code is populated in the Create Test Data area.



Example input data and result for a data method test

If the data method accepts input parameters, you need to specify them in the Test Data Method window. When developing the data method, you may find it useful to vary the inputs and repeat the test.

When you click the Test button in the Test Data Method window, the Decision Module runs the test and displays a message indicating the outcome in the Test Result section.

For a test that is successful, the Test Result section contains a link to view the response XML. If errors are found while testing, a link to view error details is available instead. In either case, a Retest option is available to run the test with modified data.

Keep the following points in mind when testing a data method:
  • You will likely receive an error instead of test results if the XML is not set up correctly.
  • You must be familiar with your object model and the XML schema that defines it, because you must include for each object and field required for the test, the associated XML elements and attributes. Fields may be represented in your XML schema as either attributes or simple elements, and your XML must be valid for your XML schema.
  • You must provide attribute and simple element values for all fields that are required by the data method you are testing, as well as any other data methods that are executed as a result of executing this one.
  • If this or other data methods use data object references that refer to other objects in your object model, you may need to add additional complex elements, beyond what is included in the initial template provided in the Create Test Data section.
