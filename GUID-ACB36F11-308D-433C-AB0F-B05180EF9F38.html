Active Context in Loop Statements

When working with loop statements, it is important to understand how the active context is determined and set within the Expression Editor. The following table describes the rules that govern active context.
Active Context
Cursor Position Active context is...
Not within a loop Context of the data method
Within object-based processing loop Context of the loop
Within data object reference-based loop Context of the data object reference
Within a multi-loop expression Context of the inner-most loop

If you specify an object that is not part of the active context path, the system does not permit it. The context of a field or data method used within a loop must be an object that is part of the active context path. For example, if the active context path is Application.creditApplication.personalApplicant, fields and data methods for both Application and Application.creditApplication.personalApplicant are permitted. A field or data method in a nested loop(s) is permitted if it is a valid field or data method name for an object that is part of the active context path for the current loop, any of the outer loops, or the data method. If a data object reference loop is nested, it is possible for the same object type to be in context multiple times, and then the innermost context will be used.

In addition, the system detects context-related errors pertaining to improper nesting of object-based processing loops. If you insert an object-based processing loop with a loop context path that is not of the form ActiveContext.descendant, a validation error is generated. For example, if the active context is Application.creditApplication .personalApplicant, any loop context that starts with Application.creditApplication .personalApplicant is considered valid, and all others are invalid:
  • Application.creditApplication.personalApplicant.applicantFinancial is valid.
  • Application.creditApplication.personalApplicant.applicantFinancial.income is valid.
  • Application.applicant is invalid.
