Events

An event is the smallest and most basic element in a business process or workflow. There are seven generic predefined events that you can use to in a workflow or a subflow. Start and stop are two events that must be present in a workflow or a subflow.

The following types of events are available while defining a workflow or a sub-flow:

Start

A start event begins a process or a sub-process. It defines where a process starts. A start event is represented by the symbol:

End

An end event ends a flow or a sub-flow. The workflow engine ends the current execution path. A workflow or a subflow can have multiple end events. An end event is represented by the symbol:

Error handler

An error handler event is used to catch any error that is thrown during the processing of a task, call-activity, or a subprocess. The workbench allows you to define the processing that must be done when an error occurs. You can assign an error handler when you define a task, call-activity, or a sub-process. An error handler is represented by the symbol:

Throw error

You can define a process to throw an error when there is an error in the processing. When an error occurs, the current path of execution ends. If an error handler is defined for the error, it will catch the error; else an exception is thrown. A throw error is represented by the symbol:

Use this in a sub-process, then use an error handler - throw error is caught at one level above, if you use this in your sub-process. Add the error handler at the process level

Time-based wait

A time-based wait event is timer that you can add at a point where the workflow or process waits for a user action to be taken. A time-based wait acts as a stopwatch and as an alarm clock. You attach this type of event to an activity and set a duration for the activity, which is how long the workflow will wait for the activity to be performed. When the execution reaches the time-based wait, a timer is started. If the human task or user activity is not performed within the interval of time specified, the timer fires. The activity is interrupted and another sequence of activities is triggered. A time-based wait is represented by the symbol:

Human task

A human task is an event that you add when the workflow requires an activity or task that involves a human. When the workflow execution reaches a human task, the workflow waits for the activity or task to be completed by the human. Once the action is performed, the workflow resumes. Covered Topics
  • Example: Adding an Assignment Event
    An assignment event assigns or sets a value to a variable or field in the application BOM. You can use an assignment event in both, sub-process and process or workflow.
  • Example: Adding an Error Handler Event
    An error handler catches an error in a task, call-activity, or sub-process and specifies how the processing should flow when the error occurs. You can add multiple error handlers in a workflow. An error handler must be connected to the error code and the construct or the node that throws the error. An error handler catches specified errors or all errors, and specifies the error handling logic and the processing flow when these errors occur.
  • Example: Adding a Throw Error Event
    A throw handler event catches an error in a a task, call-activity, or sub-process. When the workflow or business process encocunters a throw error event, the current path of execution ends and an error is thrown. Use the throw error event with an error handler event to allow the workflow exectuion to proceed along another path. If the throw error event is not connected to an error handler event, the system throws an exception. A throw error event must have an error code associated with it.
  • Example: Adding a Time-based Wait Event
    A time-based wait event is timer that you can add at a point where the workflow or process waits for a human action to be taken. Specify the processing flow if the activity is not performed within the interval of time specified.
