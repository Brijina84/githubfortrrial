How the Decision Module Works

The Decision Module provides a decision (for example, approve, decline, or investigate) for each credit application that is processed through the Decision Module web service.

Application data is passed from a calling application to the Decision Module in input XML documents. The Decision Module returns the decisions and reasons to the client application in output XML documents, and all application data is saved into a data repository for use in testing and report generation.

Information about your products and strategies is stored in a configuration repository. To provide a decision for each credit application, the Decision Module refers to the configuration in your repository and processes each application. Use the Strategy Management Workstation (SMW) to create product strategies and decision flows for each of your institutionâ€™s credit products.

Product strategies are represented graphically as specialized decision trees referencing decision flows.

Each decision flow is a directed graph consisting of decision flow steps, which may be rulesets, decision tables, decision trees, data method sequences, or score models.

Using the SMW, you can leverage standard credit constructs such as limit assignment tables and approve/decline rulesets. All SMW components may be versioned as well as reused.

The Decision Module is delivered with an extendable domain model that contains business objects and attributes (fields) commonly used in credit application processing. You can use the fields in the domain model in the components used for your credit products.

When you set up your products in the SMW, all of the settings are saved in the configuration repository in your development environment. After modifications are made or new components are created and tested, the repository is promoted to a production environment where the new configuration can be used.
